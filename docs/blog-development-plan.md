# 博客系统开发计划

本文档基于当前设计方案和原型，面向正式开发启动。技术基线为：

- 前端：Vue 3 + Vite + TypeScript + Vue Router + Pinia
- 后端：Go + Gin + Bun ORM
- 数据库：PostgreSQL，全文搜索使用 PostgreSQL `tsvector` + GIN 索引
- 缓存与任务：Redis，可配合 Asynq 或自研 Go worker
- 部署：Docker + Nginx

## 1. 当前仍需确认的不确定点

以下问题不解决也可以启动脚手架和基础模块，但建议在第一周内定下来，避免后续返工。

| 问题 | 影响 | 建议默认决策 |
| --- | --- | --- |
| 前端 UI 方案 | 决定组件封装方式、样式规范和后台开发效率 | 前台使用 Tailwind CSS + 少量自定义组件，后台使用 Naive UI，统一设计 token |
| SEO 实现方式 | 影响文章详情、专题页、归档页的搜索收录 | MVP 使用 Go 服务端输出文章详情页基础 HTML、meta、JSON-LD，Vue 负责交互增强 |
| Go 数据访问方案 | 影响开发效率、SQL 可控性和迁移方式 | MVP 使用 GORM 加速开发，复杂查询用原生 SQL；后续可对搜索和统计改为 sqlc / pgx |
| 鉴权方案 | 影响前后台权限、中间件、Cookie 安全和移动端扩展 | Web 优先使用服务端 Session + HttpOnly Cookie；API 内部用 Gin 中间件鉴权 |
| Markdown 编辑器 | 影响后台写作、投稿、预览和 XSS 清理 | MVP 使用 ByteMD 或 Milkdown，后端统一做 Markdown 渲染和 HTML 清理 |
| 文件存储 | 影响媒体库、上传、备份和部署成本 | 开发环境本地 MinIO 或本地目录，生产使用 OSS / S3 兼容对象存储 |
| 评论审核默认策略 | 影响用户体验和内容安全 | MVP 默认评论进入待审核，管理员可改为自动通过 |
| 投稿开放策略 | 影响后台审核工作量和反垃圾策略 | MVP 开放登录用户投稿，但通过频率限制和人工审核控制风险 |
| 邮件订阅 / Newsletter | 影响用户表、订阅表、邮件队列和后台统计 | MVP 只做 RSS；邮件订阅和 Newsletter 后置 |
| 后台高级页面是否首期开发 | 影响工期 | 首期只做文章、评论、用户、媒体、导航、统计、设置；导入导出、审计日志、重定向独立页后置 |

## 2. MVP 范围

### 2.1 必须交付

- 前台：首页、归档页、专题页、文章详情页、登录注册、个人中心基础页
- 内容：文章、分类、标签、封面、摘要、SEO 字段、Markdown 正文
- 后台：文章管理、编辑器、评论审核、用户管理、媒体库、导航设置、系统设置
- 用户：注册、登录、退出、邮箱验证、个人资料、我的评论、我的收藏
- 互动：评论、回复、评论点赞、文章点赞和点踩、收藏
- 搜索：PostgreSQL 全文搜索，支持标题、摘要、正文、标签加权
- 投稿：登录用户投稿、草稿、提交审核、退回修改、审核通过后发布
- 站内信：系统消息、审核消息、管理员定向消息
- 运维：Docker Compose、本地备份脚本、日志、基础限流、XSS 清理
- SEO：Sitemap、RSS、canonical、Open Graph、文章结构化数据

### 2.2 明确后置

- Newsletter 和邮件订阅
- 会员订阅、付费内容、订单
- 插件市场、主题市场
- 复杂推荐算法
- 独立搜索中间件
- 多语言内容
- Headless API 和 Webhook
- 导入导出独立管理页、审计日志独立管理页、重定向独立管理页

## 3. 推荐里程碑

### M0：项目初始化，2-3 天

目标：搭好可持续开发的基础工程。

任务：

- 建立 `web`、`api`、`deploy` 目录结构
- 初始化 Vue3 + Vite + TypeScript
- 初始化 Go + Gin 项目
- 配置 PostgreSQL、Redis、对象存储开发环境
- 配置 Docker Compose
- 配置 ESLint / Prettier、Go fmt / vet、基础 CI
- 确定环境变量规范和配置加载方式

验收：

- 本地一条命令启动前端、后端、数据库和 Redis
- 前端能请求 `/api/health`
- 后端能连接 PostgreSQL 和 Redis

### M1：数据模型和基础后端，1 周

目标：先把核心数据结构和权限边界落稳。

任务：

- 建立数据库迁移机制
- 实现用户、角色、文章、分类、标签、媒体、评论、站内信基础表
- 实现 `posts.search_vector` 和 GIN 索引
- 实现 Session、登录、注册、退出、当前用户接口
- 实现后台鉴权中间件和权限校验
- 实现统一错误响应、分页参数、审计日志基础能力

验收：

- 可以完成注册、登录、退出
- 后台接口未登录不能访问
- 文章、分类、标签基础 CRUD 能通过 API 测试
- 搜索字段能随文章创建和更新生成

### M2：内容管理后台，1-2 周

目标：后台可以创建和发布文章。

任务：

- 后台布局、侧栏、路由守卫
- 文章列表、筛选、分页
- Markdown 编辑器、预览、封面选择
- 草稿、发布、下架、定时发布字段
- 分类和标签选择
- 媒体上传、媒体列表、Alt 文本编辑
- SEO 字段编辑

验收：

- 管理员可以创建草稿并发布文章
- 发布后前台能看到文章
- 媒体资源可以上传并绑定到文章
- 文章详情 API 返回 `id`、`slug`、SEO 信息和互动计数

### M3：前台阅读体验，1 周

目标：读者可以完整浏览和检索内容。

任务：

- 首页、归档页、专题页、文章详情页
- 面包屑、返回逻辑、目录、代码块复制
- PostgreSQL 全文搜索接口和搜索弹层
- RSS、Sitemap
- Go 服务端输出文章详情基础 HTML、meta 和 JSON-LD
- Footbar 组件迁移为 Vue 组件，运行时间动态计算

验收：

- 文章详情在未登录状态可阅读
- 搜索能按标题、摘要、正文、标签返回结果
- 文章详情源 HTML 包含标题、描述、canonical、OG 信息
- RSS 和 Sitemap 可访问

### M4：用户互动，1 周

目标：登录用户可以评论、收藏、点赞和点踩。

任务：

- 评论提交、回复、我的评论
- 评论审核后台
- 评论点赞、举报
- 文章点赞、点踩、取消反馈
- 文章收藏、我的收藏
- 频率限制和禁言校验

验收：

- 未登录用户不能评论和互动
- 禁言用户不能评论、回复、投稿
- 同一用户对同一文章只能有一个 `like` 或 `dislike`
- 评论默认进入待审核，审核通过后前台展示

### M5：投稿和站内信，1-2 周

目标：完成登录用户投稿到后台审核发布的闭环。

任务：

- 投稿编辑页、草稿保存、提交审核
- 我的投稿列表、审核意见展示
- 后台投稿审核、退回、拒绝、通过发布
- 审核结果生成站内信
- 站内信收件箱、已读、归档、删除
- 后台发送站内信、群发、撤回未读消息

验收：

- 注册用户不能直接发布文章
- 投稿通过后变成正式文章
- 退回修改必须有审核意见
- 投稿审核结果在站内信和我的投稿中都能看到

### M6：设置、统计和上线准备，1 周

目标：系统具备可上线的运营和维护能力。

任务：

- 系统设置、导航设置、评论策略、投稿策略
- 基础统计：PV、UV、阅读量、热门文章、搜索词
- 备份脚本、对象存储配置
- 日志、慢查询、错误监控
- 安全检查：XSS、CSRF、上传校验、限流、Cookie 设置
- 部署脚本和生产环境配置

验收：

- 管理员可以调整站点基础配置
- 生产环境 Docker Compose 能启动
- 数据库可备份和恢复
- 关键接口有访问日志和错误日志

## 4. 开发顺序建议

建议按“后端数据和权限先行，后台内容能力其次，前台体验随后接入”的顺序推进。

1. 先做数据库迁移、鉴权、文章 CRUD。
2. 再做后台文章编辑和媒体库。
3. 然后做前台文章展示、SEO、搜索。
4. 再做用户互动、评论审核、收藏和反馈。
5. 最后做投稿审核、站内信、统计和上线能力。

这样可以减少前台先做完但没有真实接口可接的等待，也能尽早暴露数据模型问题。

## 6. 质量标准

- API 返回统一结构，错误码可被前端稳定处理
- 所有列表接口支持分页，后台列表支持筛选和排序
- 所有写接口需要鉴权，敏感操作写审计日志
- Markdown 渲染结果必须做 HTML 清理
- 上传文件必须校验类型、大小和扩展名
- 搜索接口需要记录慢查询并限制关键词长度
- 前端路由需要处理加载、空状态、错误状态和未授权状态
- 核心流程需要有集成测试：登录、发布文章、搜索、评论、投稿审核

## 7. 启动前检查清单

- 确认 UI 方案：Tailwind + Naive UI 是否接受
- 确认 Go 数据访问：GORM 作为 MVP 默认方案是否接受
- 确认 Session Cookie 方案是否满足部署环境
- 确认对象存储供应商和开发环境替代方案
- 确认生产域名、备案信息、HTTPS 和邮件服务
- 确认首期是否需要邮箱验证；如果需要，需要先配置邮件服务
- 确认是否先做 RSS，邮件订阅和 Newsletter 不进入 MVP
- 确认后台高级页面是否按后置处理

## 8. 主要风险

| 风险 | 表现 | 控制方式 |
| --- | --- | --- |
| SEO 被 SPA 影响 | 搜索引擎抓不到文章 meta 或正文 | Go 输出文章详情基础 HTML 和 meta，Vue 做交互增强 |
| 数据模型过早复杂 | 投稿、文章、版本历史重复建模 | 投稿复用文章表，通过状态、submitter_id、review 字段隔离 |
| 评论和投稿带来垃圾内容 | 后台审核压力增加 | 默认审核、频率限制、禁言、举报、关键词过滤 |
| 媒体库存储失控 | 图片占用大、引用关系不清 | 对象存储、缩略图、使用次数、删除前影响检查 |
| 全文搜索性能下降 | 文章变多后查询慢 | GIN 索引、权重字段、关键词长度限制、慢查询优化 |
| 前后台组件风格割裂 | 使用体验不统一 | 抽取颜色、间距、按钮、表单 token，后台组件套统一主题 |
